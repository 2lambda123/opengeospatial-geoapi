Open issues about OpenGIS's UML

The following are GeoAPI comments (not OGC comments):

 

General remarks

  • UML found in the 2003-073 specification contains only attributes or associations, no methods. Many services found in the legacy 2001-009 specification are not available in latest UML. More specifically:

    • There is nothing like a transform method taking a point in argument and returning a transformed point (e.g. the legacy CT_MathTransform interface).

    • There is no factory at all (e.g. the legacy CT_CoordinateTransformationFactory interface).

  • The "Well Know Text" (WKT) format described in the 2001-009 specification is not mentioned in the 2003-073 specification. The 2003-073 specification do not proposes any standard parameter name. The 2001-009 specification proposed some of them (e.g. "semi_major", "semi_minor", "standard_parallel_1"). Those parameter names are listed on the remotesensing.org web site.

  • There is no localization support in both specifications. Localization is an European Union requirement. UML should specify which CharacterString are localizable and which ones are not. A proposal is to add a new stereotype in UML: LocalizableCharacterString.

 

Remarks relative to hierarchy and attributes

  • The 2003-073 specification duplicates the name and identifier attributes in many interfaces, with different identifier name (e.g. "ellipsoidName", "datumName", "csName", "crsName", etc.). A common super-interface like the legacy CS_Info would be more convenient.

  • Interface CartesianCS looks like a special case of ObliqueCartesianCS. Maybe it should extends ObliqueCartesianCS. It would help to clarify the semantic of ImageCRS (no need for separated getCartesianCS() and getObliqueCartesianCS() methods).

  • The ImageCRS interface is defined as a kind of engineering CRS, but doesn't extend EngineeringCRS. Maybe it should (there is no obvious technical reason preventing that).

  • "operationMethod" attribute is defined in CoordinateOperation, but applied only to Transformation (a sub-interface). Maybe it should be defined only in Transformation interface.

  • "minimumOccurs" attribute is declared in GeneralOperationParameter, but "maximumOccurs" is declared in OperationParameterGroup (a sub-interface). Maybe they should be declared in the same place.

  • Some attribute names are misleading regarding the type. For example "includesCRS" in CompoundCRS is of type CoordinateReferenceSystem, not CRS (which exists as an other interface).

  • It may be worth to get ride of the CRS interface (see its javadoc for a rational).
